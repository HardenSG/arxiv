### 🎯 核心问题

LLM Agent每次遇到新任务都要从头推导执行策略！
一个调试过100次空指针异常的代码Agent，第101次还是像第一次一样从头推理😱

程序化知识随上下文窗口消失——**这就是Skill层要解决的核心问题**！

---

### 🧠 形式化定义

**Skill = (C, π, T, R)**

| 组件 | 含义 | 作用 |
|------|------|------|
| **C** | 适用性条件 | 判断技能是否适用于当前上下文 |
| **π** | 可执行策略 | 生成动作或调用其他技能 |
| **T** | 终止条件 | 判断技能何时完成 |
| **R** | 调用接口 | 程序化调用的签名 |

**为什么四元组是必须的？**
- 无C → 技能无法自我选择
- 无T → 无法组合（调用者不知何时恢复）
- 无R → 无法程序化调用
- 无π → 没有执行能力

---

### 🔄 技能 vs 相关抽象（核心区分！）

| | Tool | Plan | Memory | Skill |
|--|------|------|--------|-------|
| **复用单位** | 单API调用 | 任务分解 | 存储观测 | **程序化模块** |
| **执行** | 无状态单次 | 一次性推理 | 检索无执行 | **可调用工作流+终止** |
| **验证** | I/O模式 | 步骤一致性 | 相关性 | **结果正确性+安全** |
| **组合** | 顺序链接 | 层次分解 | 间接 | **层次/DAG/递归** |

**人话版**：
- Tool = 原子操作（搜索API）
- Plan = 一次性计划（用完就丢）
- Memory = 记忆碎片（知道"什么"）
- Skill = 可调用程序（知道"怎么做"）

### 🛠️ 七大设计模式

#### 1️⃣ 元数据驱动披露
- **代表**：Claude Code、Semantic Kernel
- **原理**：用简短元数据发现技能，选中才加载完整指令
- **优势**：Token效率高，可扩展到大型技能库
- **风险**：元数据中毒

#### 2️⃣ 代码即技能
- **代表**：Voyager、SWE-agent、CodeAct
- **原理**：技能=可执行程序（Python函数等）
- **优势**：确定性、可测试、可组合
- **风险**：API变化时脆弱

#### 3️⃣ 工作流强制
- **代表**：TDD Agents、LATS
- **原理**：强制Agent按固定流程执行
- **优势**：可靠+可审计
- **风险**：灵活性牺牲

#### 4️⃣ 自进化技能库
- **代表**：Voyager、DEPS
- **原理**：Agent自己生成+验证技能
- **优势**：适应新任务
- **风险**：⚠️ 自生成技能平均-1.3pp！

#### 5️⃣ 混合NL+Code
- **代表**：Claude Skills、ReAct
- **原理**：自然语言+代码组合
- **优势**：灵活+确定
- **风险**：边界模糊

#### 6️⃣ 元技能
- **代表**：Self-Instruct、CREATOR
- **原理**：技能创造技能
- **优势**：自动化扩展
- **风险**：递归错误放大

#### 7️⃣ 市场分发
- **代表**：OpenAI GPT Store、**ClawHub**、MCP
- **原理**：技能作为可分发包
- **优势**：社区生态
- **风险**：⚠️ ClawHavoc事件1200个恶意技能！

### 📊 SkillsBench实证（最关键！）

| 技能类型 | 效果 |
|---------|------|
| **精心设计（Curated）** | **+16.2个百分点** 🎉 |
| **自生成（Self-generated）** | **-1.3个百分点** 📉 |
| 小模型+好技能 | > 大模型无技能 |

**结论**：精心设计的技能比即时生成的可靠得多！小模型+好技能可以逆袭大模型！

### 💡 对我的启发

1. **手动设计技能库是值得的**——+16.2pp不是梦
2. **不要盲目让Agent自生成技能**——可能适得其反
3. **技能可以作为模型规模的替代**——效率倍增器
4. **安全要一开始就考虑**——市场分发模式的风险极高
5. **模式组合是常态**——实际系统常用2-4个模式
